## 1. 개요
개인 디스코드 서버의 **지정 채널**에 **등록된 사용자**가 러닝 기록 사진을 올리면, 봇이 자동 감지해
**연속 러닝 일수(스트릭)** 를 기록하고 슬래시 커맨드로 조회·랭킹·집계를 제공한다.

## 2. 운영 환경 / 배포
- 호스트: **SRV-1** (Rocky Linux 9.7, x86_64). Claude Code 세션이 srv1에서 직접 구동(맥 개발→push 전제는 폐기).
- 전부 컨테이너: `docker compose` 로 **bot**(`python:3.12-slim` + discord.py + Tesseract) + **db**(`postgres:16-alpine`).
- 봇 전용 PG는 **호스트 포트 미발행**(내부 네트워크 `internal` 전용) → 호스트의 기존 PG(127.0.0.1:5432)와 충돌 없음.
- 비밀값은 `.env`(gitignore). 슬래시 커맨드는 **길드 스코프 동기화**. 특권 intent는 **MESSAGE CONTENT만** 사용.
- `restart: unless-stopped` → 재부팅 자동 복구. 데이터는 named volume `pgdata`.
- 원격 저장소: https://github.com/Elena-Jung/running-streak (Public). `.env`·비밀·타인 데이터 미포함.
- **커밋 가드**: `.githooks/pre-commit` 가 민감 패턴(사설 IP·SSH 설정·개인키·토큰 형태)을 커밋 단계에서 차단. 새 클론은 `git config core.hooksPath .githooks` 1회 설정 필요(우회: `--no-verify`).
- 운영: `docker compose up -d --build` / `logs -f bot` / `down`. 절차 상세는 `README.md`.
- **호스트 비종속**: 설정은 전부 `.env`. SRV-1 외 제3자 서버에도 배포 가능 — 사람용 `docs/SELF_HOSTING.md`, 에이전트용 `AGENTS.md`. (SRV-1 관련 기술은 원 배포 기록.)
- **자랑 카드 정적 페이지(선택)**: `web/index.html` 은 마일스톤 카드 생성기(순수 HTML/JS, 외부요청 0). **봇/DB 컨테이너 밖**의 정적 파일로, 아무 정적 호스트/리버스 프록시가 서빙한다(러닝-스트릭 스택엔 새 서비스 추가 없음 — 여전히 bot+db 2개). 봇은 `BRAG_BASE_URL` 로 그 주소만 알면 된다. **SRV-1 배포의 실제 서빙 주소·경로는 공개 문서에 기재하지 않는다**(실값은 `.env` 의 `BRAG_BASE_URL` 에만). 서버에 저장하는 데이터는 없다(§7).

## 3. 확정 결정사항 (불변 — 함부로 되돌리지 말 것; 명세 9/12.4.5)
- **언어**: Python + discord.py. **OCR**: 로컬 Tesseract만(클라우드 OCR/비전 LLM 금지).
- **날짜 기준**: 메시지 업로드 시각을 **KST(Asia/Seoul) '러닝 하루'** 로 환산. 사진 속/OCR 날짜는 쓰지 않는다.
  · **하루 경계는 자정이 아니라 KST 04:00** (`DAY_RESET_HOUR=4`). 00:00~03:59 새벽 업로드는 **전날**로 집계(하절기 새벽 러닝 배려). 구현 `to_run_date()` = KST 변환 후 4시간 빼고 `.date()`. 한국시간은 DST 없어 안전.
  · 과거 데이터는 **소급 마이그레이션하지 않음**(전진 적용만). 〔2026-06-25 결정〕
- **유예(grace)**: 마지막 러닝 이후 **간격 ≤ 3일 유지, ≥ 4일 리셋**. **실제 뛴 날만** 카운트(유예일 미카운트).
- **하루 1회**: 같은 날 여러 번 올려도 1회만 집계.
- **조회는 읽기 전용**: 저장값을 바꾸지 않고, 표시 때만 `today−last_run`으로 유효성 보정(유령 스트릭 방지).
- **스케줄러/백그라운드 작업 금지**: 모든 계산은 *사진 이벤트* 또는 *조회 시점*에만.
- **구현 금지**: 위기 알림, 주간/월간 리포트 자동 게시, 그래프 생성, 목표·배지, 대규모 확장 설계.
  · **의도적 예외 — 자랑 카드(2026-07-03 결정)**: `/자랑` 마일스톤 카드는 '배지 금지'의 표면과 닿지만, 그 금지의 취지는 **자동·푸시형 게이미피케이션(스케줄러·자동 알림/게시)** 배제다. 자랑 카드는 **옵트인·사용자 개시(pull, 슬래시 커맨드)·자동게시 및 스케줄러 없음·서버 저장 없음** 이라 성격이 다르므로 예외로 허용한다. 위기 알림·독촉·자동 게시는 여전히 금지 — 단 **마일스톤 달성 '당일'의 완료 메시지에 `/자랑` 권유 한 줄**은 예외 범위에 포함한다(사진 이벤트 응답에 얹는 축하 문구일 뿐, 스케줄러/푸시가 아님. 2026-07-03).
- **OCR은 부가정보**: 실패해도 스트릭에 영향 없음. 신뢰도 낮아 **채널 미노출**, `run_logs`에만 저장.

## 4. 데이터 모델 (PostgreSQL)
`runners` — 사람당 스트릭 상태(O(1) 조회용 캐시):
| 컬럼 | 의미 |
|---|---|
| `user_id` (PK) | 디스코드 사용자 ID |
| `registered` | 등록(옵트인) 여부. 해제 시 FALSE(기록 보존) |
| `last_run_date` | 마지막으로 뛴 날(KST '러닝 하루' 논리 날짜, 04시 리셋) |
| `current_streak` | 마지막 뛴 날 기준 연속 일수 |
| `max_streak` | 역대 최장 |
| `total_runs` | 총 뛴 날 수 |

`run_logs` — **'뛴 날 1행' 원장(source of truth)**. 취소/재계산의 근거이자 `/기록`·이번달 집계원:
| 컬럼 | 의미 |
|---|---|
| `id` (PK) | |
| `user_id`, `run_date` | `run_date`=논리 날짜(04시 리셋). **UNIQUE(`run_logs_user_day_uq`)** → 하루 1행 강제 |
| `distance_km` NUMERIC | OCR 거리(소수만 인정) |
| `duration_sec` INT | OCR 운동시간(초) |
| `pace_sec_per_km` INT | OCR 페이스(초/km) |
| `calories` INT | OCR 칼로리 |

> **데이터 최소화**: OCR 원문(`raw_text`)은 저장하지 않는다(어떤 조회에도 안 쓰이고 타인 식별정보가 섞일 수 있어 보관 위험만 있음 → 컬럼 제거, 〔2026-06-25〕). 봇이 보관하는 개인정보는 디스코드 ID + 파생 수치 4개뿐.

> 불변식: `runners`의 (current_streak, max_streak, last_run_date, total_runs)는 `run_logs`의 run_date 집합으로부터
> `streak.recompute_from_dates()`로 **재계산 가능**해야 한다. 취소는 이 재계산을 사용한다.

## 5. 동작 로직
### 5.1 사진 업로드 집계 — `events.handle_message` ([bot/app/events.py](bot/app/events.py))
1. 봇/다른봇/DM 무시 → 채널이 `TARGET_CHANNEL_ID` 아니면 무시 → 이미지 첨부 없으면 무시 → 미등록자 무시.
2. `today = to_run_date(message.created_at)` (KST 04시 리셋: 00:00~03:59는 전날). `record = load(user)`. 조회측도 `current_run_date()`로 같은 기준 사용.
3. `compute_on_run`으로 **counted 게이트만** 판정(gap≤0이면 무음 종료, 반응도 없음). 실제 스트릭 값은 record_run 이 원장에서 재계산해 돌려준다.
4. **즉시 ✅ 반응**(접수 표시, 권한 없으면 무시).
5. OCR best-effort — **`asyncio.to_thread`로 워커 스레드 오프로드 + Semaphore(2)로 동시 수 제한** → 이벤트 루프·하트비트·다른 커맨드를 막지 않는다.
6. `record_run(...)` → `(recorded, current_streak)`. 트랜잭션 안에서 `SELECT … FOR UPDATE`로 같은 사용자를 직렬화하고 run_logs INSERT 후 **원장 전체 재계산**으로 runners 갱신.
   - `recorded=False`(같은 날 유니크 충돌) → **무음 종료**. record_run 이 예외면 ✅를 ⚠️로 바꿔 **거짓 접수 방지**.
7. 채널에 `### 러닝 기록 완료. N일째 연속입니다.` + 업로더 멘션 전송. **완료 메시지는 매번 간결하게**(04시 경계 안내는 넣지 않음). 단 **N 이 마일스톤(10 배수)인 당일**에만 `🎉 연속 N일 달성! /자랑 …` 권유 한 줄 추가(§3 예외).
   - 04시 경계(새벽 러닝=전날) 설명은 **`/스트릭`·`/캘린더` 출력의 subtext로 이동**(매 업로드마다 노출하지 않기로 결정, 2026-06-25).
   - **소프트 힌트**: OCR 4필드를 모두 못 읽었으면(엉뚱한 이미지 가능성) 메시지 끝에 `/달리기 취소` 안내를 subtext로 덧붙인다. **집계는 막지 않는다**(명세 2: OCR이 스트릭을 좌우하지 않음). 사진형 정상 러닝(신발·워치·트레드밀·해외앱)도 OCR이 비기 쉬워 게이트/확인은 두지 않기로 결정.

### 5.2 스트릭 규칙 — `streak.compute_on_run` ([bot/app/streak.py](bot/app/streak.py))
- `last_run is None` → `(1, True)` (첫 러닝).
- `gap = (today − last_run).days`. `gap ≤ 0` → `(current, False)`(오늘 이미/시계이상, 무시).
- `1 ≤ gap ≤ 3`(GRACE_MAX_GAP=3) → `(current+1, True)`. `gap ≥ 4` → `(1, True)`(리셋).

### 5.3 조회(읽기 전용) — `streak.effective_streak`
- `gap > 3` → `0`(유령 스트릭 차단), 아니면 저장된 current. **저장값 변경 없음.**

### 5.4 기록/취소 재계산 — `db.record_run` · `db.undo_last_run` + `streak.recompute_from_dates`
- **record_run·undo 모두 `SELECT … FOR UPDATE`로 사용자 단위 직렬화** → load→계산→저장 경합(TOCTOU)·동시 기록/취소 경합 제거.
- 둘 다 INSERT/DELETE 후 남은 run_date 집합으로 **runners 전체 재계산**(증분 아님) → 재등록 후 유령 부활·UPDATE-0행 문제도 함께 사라짐. runners 는 항상 원장의 함수.
- `recompute_from_dates(dates)`: 정렬·중복제거 후 동일 유예 규칙으로 (current, max, last, total) 산출.

### 5.5 OCR 부가정보 — [bot/app/ocr.py](bot/app/ocr.py)
- 전처리: 흑백+autocontrast(A), **원본 컬러**(C, 컬러 글씨), **적응형(국소 평균 대비) 이진화 2장**(B, 밝은/어두운 글씨). 각 변형 PSM 6/11 후 합쳐 파싱.
  · **적응형 = 위치 크롭·고정 임계값 없음**: 각 픽셀을 주변 평균과 비교해 사진 위 흰 히어로·카드 위 어두운 글씨를 **화면크기·앱·비율 무관**하게 분리(블러 반경은 짧은 변 비례 → 해상도 무관). 기존 '상단 크롭' 방식의 취약점 제거.
- 칼로리 2단 그리드(런데이 등 값/라벨이 다른 줄): 텍스트 실패 시 `image_to_data` 좌표로 '칼로리' 라벨과 같은 열의 가까운 숫자를 택함(`_calorie_from_layout`).
- 운동시간: **날짜/업로드시각 줄 제외**(`2026년 6월 10일 (수) 13:35` 등) → 날짜의 시각을 운동시간으로 오인하지 않음.
- 거리: 단위 인접 정규식 → 폴백(독립 소수, 시간/페이스/칼로리/`/km` 제외). **정수 단독 제외**(소수만 인정), **`km/h`(평균 속도)는 거리로 잡지 않음**(메인·폴백 모두).
- 칼로리: 상한 20000(OCR 노이즈 컷). 거리/속도 혼동 방지는 `test_ocr.py`로 회귀 보호.
- **백계산 엔진(거리/속력/시간 역산)**: OCR 이 놓친 필드만 다른 값으로 역산한다(**OCR 값은 절대 덮어쓰지 않음 · None 만 채움 · 유도값은 범위검사 통과분만 저장**). 우선순위 — ① 페이스 없으면 거리·시간(코어)으로 `pace=시간/거리`(가장 정확), 없으면 속도로 `pace=3600/(km/h)`(폴백), ② 거리 없으면 `시간/페이스`, ③ 시간 없으면 `거리×페이스`. 범위: 거리 0<d≤100, 페이스 120~1800, 시간 30~86400.
- **삼성헬스 트레드밀(헬스장) 화면**: 페이스 대신 `평균 속도 N km/h` 를 표시 → `extract_speed_kmh`로 읽어 페이스로 환산(역산 입력 전용). **속도 DB 컬럼은 없음** — 페이스로만 환산해 스키마·마이그레이션·`/스트릭` 페이스 표시 무변경. 거리·시간이 둘 다 있으면 속도(반올림 표시값)보다 코어 유도를 우선.
- 시간: `HH:MM:SS`/`MM:SS`, **상태바 시계 줄(KT/%/AM 등) 제외**, **뒤에 `/km` 붙은 페이스는 제외**, 후 최장 후보.
- 페이스: `분'초"`(6'00") 우선, 없으면 **콜론형 `5:30/km`**(`/km` 필수). 120~1800초/km. 칼로리: `N kcal`/`칼로리` 양방향, 상한 20000.
- 이미지 25MB 초과 시 OCR 생략(집계는 정상). 입력 포맷: PNG/JPG/WEBP/GIF/BMP(HEIC 제외).
- DB 접속 DSN은 유저/비번을 URL 인코딩(특수문자 비번도 안전).

### 5.6 텍스트 시각화 — [bot/app/charts.py](bot/app/charts.py) (순수 함수)
- `sparkline(values)`: 유니코드 블록(▁▂▃▄▅▆▇█) 추세. 페이스는 작을수록 빠르므로 호출자가 부호를 뒤집어 넘겨 개선=우상향으로 보이게 함.
- `month_calendar(year, month, run_days)`: ASCII 월 달력(코드블록용, 일요일 시작, 달린 날 `*`).
- **명세 9번 '그래프 생성' 회피**: matplotlib 등 이미지 렌더링 없이 메시지 텍스트로만 그림. 의존성·스케줄러 추가 없음.
- **명세 '리포트 자동게시' 회피**: `/캘린더`·주/월 합계는 사용자가 부를 때만 계산하는 조회(자동게시 아님).

## 7. 무결성 안전장치 & 점검 결과
- **동시성**: record_run·undo·**purge_user** 가 runners 행 `FOR UPDATE`로 사용자 단위 직렬화 → 같은 날 경합은 UNIQUE 로 하나만 성공, 서로 다른 날 동시 업로드(TOCTOU)는 직렬 재계산으로 최종 일관(라이브 동시성 테스트 통과). **OCR 은 워커 스레드 오프로드라 집계·커맨드·하트비트를 막지 않음**.
- **일관성**: `record_run`/`undo_last_run`/`purge_user` 모두 트랜잭션 → runners·run_logs 동시 갱신/롤백.
- **자원 보호(폭주 방어)**: 다운로드+OCR 을 `_HEAVY_SEMAPHORE(4)`로 묶어 동시 25MB 적재 상한(OOM 방지), 사용자별 **쿨다운 3초**로 한 명의 연속 업로드 독점 차단. (당일 재업로드는 gap≤0 으로 다운로드·OCR 전에 이미 컷.)
- **킬스위치**: `BOT_PAUSED=true` → 집계 즉시 중단(조회 커맨드는 동작). 잘못된 집계를 급히 멈출 때.
- **유령 스트릭**: 조회 시 `effective_streak`로 보정, 저장값 불변.
- **OCR 격리**: 모든 OCR 경로 예외 흡수, 실패해도 스트릭 무영향. 거리 오인식(정수 노이즈)은 가드로 차단.
- **거짓 접수 방지**: 저장 실패 시 ✅→⚠️, 같은날 중복 거부 시 봇 자신의 ✅ 제거(자기 리액션이라 Manage Messages 불필요).
- **시계/역전 방어**: `gap ≤ 0`이면 무시.
- **권한 의존**: 봇이 대상 채널에서 *보기/보내기/기록보기/반응추가* 권한 없으면 동작 안 함(운영 메모).
- **자랑 카드 개인정보(2026-07-03)**: 서버에 **새로 저장하는 데이터 없음**(새 DB 컬럼·업로드 저장소 없음). 배경 **사진은 브라우저 밖으로 나가지 않고**(파일 업로드 아님, `objectURL` 로컬 처리), 통계는 URL **프래그먼트(#)** 에만 실려 **서버로 전송조차 안 됨**(웹서버/프록시 접근 로그에 미기록). 링크는 **ephemeral(본인만)**. 저장물이 없으므로 `/달리기 전체삭제` 에 새 삭제 의무가 생기지 않음. 카드 표시 이름은 본인 display_name(편집·삭제 가능).

- **좀비 게이트웨이 연결(컨테이너는 정상인데 커맨드 무응답)**: 네트워크가 흔들린 뒤 discord.py 가 `RESUMED` 로그를 남기고도 실제로는 이벤트를 못 받는 half-open 상태가 될 수 있다. 증상은 **커맨드 무응답 + 로그 완전 침묵**(커맨드가 봇에 닿았다면 전역 에러 핸들러가 반드시 기록하므로, 에러조차 없으면 이벤트 미수신을 의심). 컨테이너 상태·로그의 '정상' 표시로는 안 잡힌다. 조치는 `docker compose restart bot`(새 세션 강제). 〔2026-07-19 발생·확인〕
- **DNS 고정(2026-07-19)**: 호스트 리졸버(NetworkManager)가 네트워크 재접속마다 `/etc/resolv.conf` 를 다시 써서, 전환 구간에 컨테이너가 빈/낡은 업스트림을 물면 `Temporary failure in name resolution` 으로 게이트웨이가 끊겼다(1주에 3회 발생). compose 의 bot 서비스에 `dns: [1.1.1.1, 8.8.8.8]` + `dns_opt` 를 명시해 **호스트 리졸버 상태와 분리**(봇은 공개 도메인만 조회하므로 사설 DNS 불필요). 단 **회선 자체가 끊기는 경우까지 막지는 못한다** — 그때는 위의 유실 복구 절차를 쓴다.
- **게이트웨이 단절 중 업로드는 유실됨(이벤트 리플레이 없음)**: 호스트 네트워크/DNS 장애로 봇이 디스코드에서 끊긴 사이의 업로드는, 재접속이 새 세션이면 이벤트가 재전달되지 않아 집계가 누락된다(스케줄러 금지 원칙상 자동 백필 없음). 복구 절차: 채널 히스토리를 REST 로 조회해 봇 반응(✅) 없는 이미지 메시지를 찾고, **기존 파이프라인 그대로 수동 재처리**(counted 게이트 → ✅ 반응 → 동일 OCR → `record_run` → 완료 메시지를 원본에 답장). 〔2026-07-10 DNS 장애 8분간 업로드 1건 유실 → 이 절차로 복구〕
- **공개 히스토리의 타인 데이터 처리 원칙(2026-06-25 결정)**: 과거 누출(타인 닉네임 1건+author 이메일)은 `filter-branch` 스크럽만으로는 GitHub **고아 커밋(옛 SHA 직링크)** 이 GC 전까지 남을 수 있다(강제푸시의 본질적 한계). 따라서 **레포를 삭제 후 재생성(delete+recreate)** 하여 옛 SHA까지 제거했다 — 로컬의 클린 히스토리를 새 레포에 푸시. **타인 식별정보가 공개 히스토리에 들어가면 강제푸시가 아니라 레포 재생성이 원칙**(과거 타인 러닝데이터 사례와 동일). author 이메일은 noreply 로 치환됨.

## 9. 검증 방법
- 단위 테스트: `docker compose run --rm bot python -m pytest -q` (현재 90건: 스트릭·재계산·OCR·차트·config·이벤트핸들러·04시 경계·트레드밀 속도/역산·킬스위치·쿨다운·**자랑 카드(highest_milestone·build_brag_url·완료 메시지 권유)**).

- 동시성/되돌리기: 가짜 user_id로 `record_run`/`undo_last_run`/race 스크립트 검증(일회성).
