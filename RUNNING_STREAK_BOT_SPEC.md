# 러닝 스트릭 디스코드 봇 — 구현 명세서 (Implementation Spec)

> **이 문서를 읽는 LLM에게**: 이 문서는 사람과의 설계 대화를 통해 확정된 결정들의
> 인계 문서다. 각 결정에는 **근거(WHY)** 와 **비목표(하지 말 것)** 가 함께 적혀 있다.
> 결론만 보고 "더 좋게" 바꾸려 하지 말 것. 근거가 박힌 결정은 의도된 것이며,
> 되돌리면 설계 정합성이 깨진다. 변경이 필요하다고 판단되면 구현 전에 사용자에게 먼저 확인할 것.
> 이 문서는 **1단계(아이디어 구체화)의 산출물**이다. 디렉토리 구조·테스트·배포 세부는
> 2단계(실행 계획)에서 다루며, 이 문서의 범위가 아니다.

---

## 0. 프로젝트 한 줄 요약

개인 디스코드 서버의 특정 채널에 **등록된 사용자**가 러닝 기록 사진을 올리면,
봇이 이를 자동 감지해 **연속 러닝 일수(스트릭)** 를 기록하고, 명령어로 조회·랭킹을 제공한다.

---

## 1. 런타임 / 운영 환경 (확정)

| 항목 | 값 | 비고 |
|------|-----|------|
| 호스트 | SRV-1 (Rocky Linux 9) | 홈 서버 |
| 구동 방식 | **컨테이너 (Docker)** | 봇 + DB 각각 컨테이너, compose로 묶음 |
| 외부망 정책 | **최소화** | 가능한 한 로컬에서 자기완결적으로 동작 |

**근거(WHY)**: SRV-1을 베어메탈 직접 설치 → 컨테이너 격리 방식으로 전환 중이다.
호스트의 기존 베어메탈 PostgreSQL에는 의존하지 않고, 봇 전용 DB를 컨테이너로 띄운다.
따라서 이 봇도 처음부터 컨테이너 철학을 따른다.

**비목표**: 기존/외부 PostgreSQL 인스턴스에 얹지 말 것. 봇 전용 DB를 컨테이너로 띄운다.

## 2. 데이터 추출 (OCR)

- 사진에서 **페이스·거리 등은 로컬 Tesseract OCR로 시도**하되, 이는 **부가정보**다.
- OCR이 실패해도 **스트릭 집계는 영향받지 않아야 한다** (아래 3번 날짜 기준 참고).

**근거(WHY)**: 러닝 기록 사진은 앱(삼성헬스/RunDay 등)마다 레이아웃·색상·폰트·날짜 포맷이
제각각이라 단일 앱 특정이 불가능하다. 트랙 배경 위 컬러 텍스트 등은 Tesseract가 깨지기 쉽다.
따라서 OCR 결과에 핵심 로직(스트릭)을 의존시키지 않는다.

**비목표**:
- OCR로 읽은 **날짜를 스트릭 기준으로 쓰지 말 것** (날짜는 3번 참조).
- 외부 클라우드 OCR(CLOVA 등)·비전 LLM API를 기본 경로로 쓰지 말 것 (외부망 최소화).
- OCR 정확도를 높이려고 과도한 전처리 파이프라인에 시간을 쏟지 말 것 — 부가정보일 뿐이다.
- **폴백 라인**: OCR이 지속적으로 불안정하면, 페이스·거리 추출을 포기하고
  "스트릭만 기록하는 형태"로 단순화해도 된다. 이는 사용자가 명시한 허용된 후퇴다.

## 3. 날짜 기준 (스트릭의 척추 — 확정)

스트릭 판정의 날짜는 **메시지 업로드 시각(디스코드 메시지 타임스탬프)** 을 기준으로 한다.

**근거(WHY)**: 스트릭은 *날짜의 정확성*이 생명인데, 업로드 시각은 OCR과 무관하게
100% 안정적인 유일한 진실원이다. 사진 속 날짜에 의존하면 OCR이 깨질 때 스트릭 전체가 흔들린다.

**주의**: 업로드 시각의 "날짜"는 **사용자 타임존(Asia/Seoul, KST) 기준 '러닝 하루'**로 환산해야 한다.
UTC 자정 경계로 날짜가 하루 밀리면 안 된다.

**하루 경계 = KST 04:00** (자정 아님): 00:00~03:59(KST) 새벽 러닝은 **전날**의 러닝으로 집계한다.
밤늦게·새벽에 뛰는 사용자(특히 하절기)를 배려하기 위함이다. 즉 KST로 환산한 시각에서 4시간을
뺀 날짜가 그 러닝의 "날짜"다(`to_run_date`). 한국시간은 DST가 없어 안전하다.

## 4. 등록제 (옵트인)

- 사용자는 `/달리기 등록` 으로 **선수 등록(옵트인)** 한다.
- **등록된 사용자가 지정 채널에 올린 이미지만** 자동 집계 대상이다.
- 등록 안 한 사용자의 사진, 또는 다른 채널의 사진은 **무시**한다.
- `/달리기 해제` 로 등록을 취소할 수 있다.

**근거(WHY)**: 채널의 모든 이미지를 무조건 집계하면 풍경·밈 사진까지 오탐된다.
등록제를 두면 봇은 *등록된 user_id 집합*만 보면 되어 오탐이 줄고 데이터가 사람 단위로 깔끔해진다.
등록은 **한 번만 누르는 스위치**이며, 이후엔 그냥 사진만 올리면 자동 집계된다(매번 명령어 X).

**비목표**: 매 러닝마다 명령어로 선언하게 만들지 말 것. 등록은 1회성 옵트인이다.

## 5. 스트릭 규칙 (확정 — 가장 중요)

### 5.1 유예(grace) 방식
- 단순 리셋(하루만 빠져도 0)이 **아니다**.
- 마지막 러닝 이후 **간격이 3일 이내면 스트릭 유지**, **4일 이상이면 리셋**.

```
마지막 러닝 = 월요일 기준
─────────────────────────────────
화요일에 뜀  → 간격 1일 → 유지 ✓
수요일에 뜀  → 간격 2일 → 유지 ✓
목요일에 뜀  → 간격 3일 → 유지 ✓  (3일까지 봐줌)
금요일에 뜀  → 간격 4일 → 리셋 ✗
─────────────────────────────────
규칙: 간격 ≤ 3 이면 유지, 간격 ≥ 4 이면 리셋
```

### 5.2 카운트 방식
- 스트릭 숫자는 **실제로 뛴 날만 카운트**한다.
- 유예로 건너뛴(안 뛴) 날은 **카운트를 올리지 않는다**. 유예는 "끊김을 막는 보호막"일 뿐
  카운트를 올리는 장치가 아니다.
- 예: 월요일 1일째 → (화·수 쉼) → 목요일 다시 뜀 → **2일째** (4일째가 아님).

**비목표**:
- 달력상 경과일로 카운트하지 말 것 (안 뛴 날을 숫자에 포함 X).
- 유예 일수를 스트릭 숫자에 더하지 말 것.

## 6. 데이터 모델 (최소 저장)

사람마다 **두 값만** 저장하면 스트릭이 굴러간다:

```
선수별 저장 데이터
────────────────────────────────────────
user_id         : 디스코드 사용자 ID (등록 식별자)
last_run_date   : 마지막으로 뛴 날짜 (KST 기준 date)
current_streak  : 현재 연속 카운트 (실제 뛴 날 기준)
────────────────────────────────────────
+ 등록 여부 자체도 저장 (등록 테이블 또는 플래그)
+ (선택) OCR로 뽑은 페이스/거리 등 부가정보는 별도 기록 가능하나 스트릭 로직과 분리
```

**근거(WHY)**: 스트릭 계산에 전체 러닝 이력을 훑을 필요가 없다. `last_run_date` 하나와
비교하면 끝이라, 데이터가 아무리 쌓여도 조회 비용이 일정(O(1))하다.

## 7. 핵심 로직 (의사코드)

### 7.1 새 러닝 사진이 올라왔을 때 (이벤트 시점에만 계산)

```
on_image_in_target_channel(user_id, message_timestamp):
    if user_id not in 등록된_선수:
        return  # 무시

    today = message_timestamp 를 KST '러닝 하루'로 환산 (경계 04시: 00:00~03:59는 전날)
    record = load(user_id)

    gap = today - record.last_run_date   # 일(day) 단위 차이

    if gap == 0:
        return  # 오늘 이미 기록됨 → 하루 1회만, 무시 (메시지도 보내지 않음)
    elif 1 <= gap <= 3:
        record.current_streak += 1       # 유예 안에서 유지
    else:  # gap >= 4
        record.current_streak = 1        # 리셋, 오늘부터 새로

    record.last_run_date = today
    save(record)

    send_message(f"### 러닝 기록 완료. {record.current_streak}일째 연속입니다.")  # 현행 문구
```

> **첫 러닝(기록 없음)** 인 경우: `current_streak = 1`, `last_run_date = today` 로 시작.
> `gap` 계산 시 last_run_date 가 없으면 첫 러닝으로 처리.

### 7.2 스트릭 조회 `/스트릭` (읽기 — 저장값 변경 금지)

```
on_command_streak(user_id):
    record = load(user_id)
    if record 없음: "아직 기록이 없습니다" 안내

    gap = today(KST) - record.last_run_date
    if gap >= 4:
        효력_스트릭 = 0   # 이미 끊김 (유령 스트릭 방지)
    else:
        효력_스트릭 = record.current_streak

    # 저장값(record)은 절대 수정하지 않는다 (조회는 부수효과 없음)
    display(효력_스트릭, 누적 통계 등)
```

**근거(WHY)**: 마지막 러닝 후 오래 지나 조회하면 저장된 current_streak은 옛날 값이라
거짓("유령 스트릭")이 된다. **보여줄 때만** `today - last_run_date`로 유효성을 덧씌워
진실을 반영한다. 단, **조회가 저장값을 바꾸면 안 된다**(읽기에 부수효과 금지).

### 7.3 누적 통계 (`/스트릭`에 함께 표시)
- 총 러닝 일수, 최장 스트릭(역대 최고), 이번 달 횟수 등.
- 최장 스트릭을 표시하려면 `max_streak` 같은 값도 저장에 추가 (current 갱신 시 함께 갱신).

### 7.4 리더보드 `/리더보드`
- 모든 등록 선수의 (효력) 스트릭을 7.2 방식으로 각자 계산해 정렬·표시.
- 소수 인원이라 전원 계산해도 부담 없음.

## 8. 명령어 세트 (MVP)

| 명령어 | 동작 |
|--------|------|
| `/달리기 등록` | 선수 등록 (옵트인). 이후 지정 채널 사진 자동 집계 시작 |
| `/달리기 해제` | 등록 취소 |
| `/스트릭` | 내 현재 연속 일수 + 누적 통계 조회 (읽기 전용) |
| `/리더보드` | 등록 선수들의 스트릭 랭킹 |

**자동 동작(명령어 아님)**: 등록된 선수가 지정 채널에 사진 업로드 시
→ 스트릭 갱신 + `### 러닝 기록 완료. N일째 연속입니다.` 텍스트 메시지.

## 9. 명시적 비목표 (이번 MVP 범위 밖 — 구현하지 말 것)

이 항목들은 "복잡도 대비 가치" 판단으로 **의도적으로 제외**했다. 임의로 추가하지 말 것.

- ❌ **스케줄러 / 매일 도는 백그라운드 작업** — 모든 계산은 "사진 이벤트" 또는 "조회 시점"에만.
      매일 자정 결석 체크 같은 시간기반 트리거를 넣지 말 것. (간격은 다음 러닝 때 거슬러 계산됨)
- ❌ **스트릭 위기 알림** (예: "오늘 안 뛰면 끊김" 자동 발송) — 스케줄러가 필요해 제외.
- ❌ 주간/월간 리포트 자동 게시.
- ❌ 그래프 이미지 생성 (matplotlib 등).
- ❌ 목표 설정·달성 배지.
      · **의도적 예외 — 자랑 카드(2026-07-03)**: `/자랑` 마일스톤 공유 카드는 허용한다. 이 금지의 취지는 *자동·푸시형 게이미피케이션(스케줄러·자동 알림/게시)* 배제였는데, 자랑 카드는 **옵트인·사용자 개시(pull, 슬래시 커맨드)·자동게시 및 스케줄러 없음·서버 저장 없음**이라 성격이 다르다. 새 게임 루프·독촉 알림은 여전히 금지. (설계 근거는 DESIGN §3.)
- ❌ 대규모(수백 명+) 확장 대비 설계 — 소수(개인 서버) 전제. 과한 인덱싱/샤딩 불필요.

## 12. 개발 / 배포 워크플로우 (확정)

이 프로젝트는 **사용자가 거의 손대지 않고 LLM(Claude Code)에 위임**하여 진행한다.
구현은 2단계에서 Claude Code 세션이 이 명세서를 읽고 착수한다.

### 12.1 역할 분리 (확정)

```
┌──────────────────────────────────────────────────────────────┐
│  개발: 맥 (로컬 머신)                                          │
│   - Claude Code를 맥북의 프로젝트 폴더에서 실행                 │
│   - 코드 / Dockerfile / docker-compose 작성, git 관리          │
│                          │                                     │
│                          ▼  git push                           │
│  배포: SRV-1 (Rocky 9)                                         │
│   - 최종 컨테이너 구동(봇 + PG)은 SRV-1에서                     │
│   - 원격 실행(배포)은 *사용자가 게이트를 쥔다* (수동 승인)      │
└──────────────────────────────────────────────────────────────┘
```

**근거(WHY)**: 맥북에 이미 개발 환경이 갖춰져 있고 git 경험이 있어, 코드는 맥에서 만들고
SRV-1엔 컨테이너만 올리는 분리가 자연스럽다. Claude Code는 기본적으로 *실행된 머신의 로컬 셸*
에서 동작하므로, 맥에서 띄우면 맥 안에서 작업한다.

### 12.2 원격(SRV-1) 실행 정책 (확정)

- Claude Code가 SRV-1로 **자동으로 SSH 진입하지 않는다** (기본 동작 아님).
- SRV-1을 실제로 건드리는 순간(배포·컨테이너 구동)은 **사용자가 수동으로 승인/실행**한다.
  필요 시 Claude Code에 SSH 명령을 그때그때 시키는 형태는 가능하되, 사람이 게이트를 쥔다.
- 반자동(SSH MCP 등으로 승인 흐름을 거쳐 원격까지 닿게)으로의 격상은 **나중 옵션**이며,
  아래 선행 조건을 마친 뒤에 검토한다.

**비목표(지금 단계)**: 원격 실행 권한을 도구에 위임하는 반자동 셋업을 지금 구성하지 말 것.

### 12.3 선행 조건 (배포 전 권장)

- 배포가 SRV-1과 원격으로 오가므로, **원격 접속 보안(키 인증·방화벽 등)을 먼저 마치는 것을 권장**한다.

### 12.4 Claude Code 착수 절차

1. 맥에 프로젝트 폴더 생성 (예: `running-streak-bot`).
2. 이 파일(`RUNNING_STREAK_BOT_SPEC.md`)을 폴더 안에 둔다.
3. 그 폴더에서 Claude Code 실행.
4. 첫 지시 예: "이 폴더의 RUNNING_STREAK_BOT_SPEC.md를 읽어줘. 11번 섹션의 2단계
   항목부터 같이 정하자. 먼저 언어·라이브러리 선택지를 제안해줘."
5. 구현 중 LLM이 9번(비목표)의 결정을 되돌리려 하면, 9번을 근거로 막는다.

> 참고: Claude Code의 정확한 설치 명령·첫 인증 절차는 버전에 따라 바뀔 수 있으므로,
> 막히면 공식 문서(docs.claude.com)를 확인한다.
